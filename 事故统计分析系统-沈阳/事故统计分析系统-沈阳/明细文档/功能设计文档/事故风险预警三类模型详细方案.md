# 事故风险预警三类模型详细方案

## 1. 方案定位

本文用于补充 [事故风险预警模块功能设计.md](c:\Users\zhaodawen\Desktop\产品工具\评估文档\事故统计分析系统-沈阳\明细文档\功能设计文档\事故风险预警模块功能设计.md) 中“趋势外推法 / 相似度匹配法 / 因素关联法”3 个预警模型的具体设计方案。

设计目标：

- 满足政务场景下“可解释、可配置、可回溯”的预警要求
- 与现有产品规划保持一致：一期先上线 `趋势外推法`、`相似度匹配法`，二期上线 `因素关联法`
- 同时支撑 `总量预警`、`行业预警`、`区域预警` 三个业务维度

---

## 2. 总体设计原则

### 2.1 模型设计原则

1. **先稳后准**：一期优先采用稳健、解释性强的规则统计模型，不直接上黑盒模型。
2. **统一输出**：三个模型虽然算法不同，但最终都要输出统一格式：`风险分值`、`预警等级`、`原因解释`、`置信度`。
3. **维度复用**：同一个模型框架可复用于总量、行业、区域三个维度，只是在样本粒度和阈值上做差异化配置。
4. **允许降级**：行业或区域样本过少时，模型自动降级，不强行产出高置信度结论。
5. **解释优先**：每条预警必须能拆出“历史规律 + 近期变化 + 外部因素/相似案例”三段原因。

### 2.2 一期 / 二期边界

| 模型 | 上线阶段 | 是否必须 | 主要数据来源 |
|------|----------|----------|--------------|
| 趋势外推法 | 一期 | 是 | 历史事故时间序列 |
| 相似度匹配法 | 一期 | 是 | 历史事故时间序列 + 历史分布特征 |
| 因素关联法 | 二期 | 否 | 事故历史 + 气象/节假日/生产负荷等外部数据 |

结论：

- 一期不要把模型 3 设计成依赖当前系统没有的数据，否则无法落地。
- 二期模型 3 应作为增强模型接入，而不是推翻一期模型体系。

## 3. 统一预警引擎设计

## 3.1 统一计算流程

```text
数据接入与治理完成
    ↓
生成预警特征快照（按日/周/月聚合）
    ↓
按维度并行计算
  ├─ 总量维度：全市
  ├─ 行业维度：道路运输 / 建筑施工 / 化工 / 商贸制造 ...
  └─ 区域维度：各区县
    ↓
每个维度分别运行模型1/模型2/模型3（模型3二期）
    ↓
模型输出标准化分值（0-100）
    ↓
多模型融合生成最终风险分值
    ↓
根据阈值映射四色预警
    ↓
生成结构化原因解释
    ↓
进入预警监控面板与处置流程
```

### 大白话理解

这整套流程其实可以简单理解成：

1. 先把事故数据整理好
2. 分别看全市、行业、区域这三类对象
3. 跑 3 种不同的判断方法
4. 每种方法都给一个分数
5. 把分数合起来
6. 再把最后结果翻译成红橙黄蓝

所以它不是“模型直接报红色预警”，而是“模型先打分，系统再按分数给颜色”。

## 3.2 统一输入数据

### 事故基础数据

来自现有事故库：

- 事故发生时间
- 所属区域
- 所属行业
- 事故类型
- 死亡人数、重伤人数、轻伤人数
- 事故等级
- 直接经济损失

### 聚合特征数据

由系统每日离线生成：

- 按日聚合：事故起数、死亡人数、重伤人数
- 按周聚合：事故起数、死亡人数、异常事故数
- 按月聚合：事故起数、死亡人数、重点行业占比、重点区域占比
- 分维度汇总：行业分布、区域分布、事故类型分布
- 历史基线：近 3 年同期均值、标准差、分位数阈值

### 外部数据（二期）

- 气象：降雨、暴雪、大风、高温、低温、雷电
- 节假日：法定节假日、重大活动、春运、开复工时段
- 生产负荷：待甲方确认
- 隐患数据：若后续接入，可作为增强因子；一期默认不依赖

## 3.3 统一输出字段

| 字段 | 说明 |
|------|------|
| `warn_dimension` | 总量 / 行业 / 区域 |
| `warn_target` | 全市 / 建筑施工 / 铁西区 等 |
| `forecast_window` | 未来 7 天 / 未来 30 天 |
| `risk_score` | 最终风险分值，0-100 |
| `warn_level` | 红 / 橙 / 黄 / 蓝 / 无预警 |
| `model_scores` | 各模型单独分值 |
| `confidence` | 高 / 中 / 低 |
| `reason_json` | 结构化原因解释 |
| `evidence_json` | 关键证据指标 |

## 3.4 建议运行频率

| 维度 | 计算频率 | 主预测窗口 | 辅助窗口 |
|------|----------|------------|----------|
| 总量预警 | 每日凌晨 2:00 | 未来 7 天 | 未来 30 天 |
| 行业预警 | 每日凌晨 2:00 | 未来 30 天 | 未来 7 天 |
| 区域预警 | 每日凌晨 2:00 | 未来 30 天 | 未来 7 天 |

说明：

- 总量维度短期波动更敏感，优先看 7 天。
- 行业和区域维度样本更稀疏，优先看 30 天，7 天作为快速抬升补充。

## 4. 模型一：趋势外推法

## 4.1 模型目标

基于历史事故时间序列，预测未来一段时间的事故总量是否明显高于历史正常水平。

这是一期最核心、最稳妥的模型，适合：

- 总量预警
- 行业预警
- 区域预警

## 4.2 设计思路

趋势外推法不建议直接采用复杂时序模型，而建议采用“`近期趋势 + 同期季节性 + 长期基线`”的混合预测方法。

原因：

- 当前业务样本量有限，尤其是行业和区域维度，复杂模型容易不稳定
- 政务场景更需要能解释“为什么判断风险升高”
- 混合预测法更容易配置阈值，也方便和甲方专家共同校准

### 大白话理解

这个模型最像人平时会说的话：

`最近事故有点往上涨，照这个势头下去，后面可能更危险。`

它最关心的不是复杂数学，而是这几件事：

- 最近有没有变多
- 比前一段时间是不是更高
- 和去年这个时候比，是不是偏高
- 最近有没有较大事故把风险拉高

## 4.3 输入特征

以某一预警对象 `X` 为例（全市 / 某行业 / 某区域）：

- `recent_7_avg`：最近 7 天日均事故起数
- `recent_30_avg`：最近 30 天日均事故起数
- `prev_30_avg`：再往前 30 天日均事故起数
- `same_period_avg_3y`：近 3 年同期平均值
- `same_period_std_3y`：近 3 年同期标准差
- `recent_death_30`：最近 30 天死亡人数
- `major_accident_flag_90`：最近 90 天是否发生较大及以上事故

## 4.4 预测公式建议

### 总量维度

未来 7 天预测值：

```text
pred_7d =
0.45 * recent_7_avg * 7
+ 0.35 * same_period_avg_3y_7d
+ 0.20 * trend_adjustment
```

中文公式理解：

`未来 7 天预测值 = 最近 7 天的平均事故水平 × 45% + 过去 3 年同期的平均水平 × 35% + 最近上涨趋势补偿 × 20%`

其中：

```text
trend_adjustment = max(0, recent_7_avg - prev_30_avg) * 7
```

中文公式理解：

`最近上涨趋势补偿 = max(0，最近 7 天日均事故起数 - 再往前 30 天日均事故起数) × 7`

解释：

- 如果最近 7 天比前面 30 天更高，就说明有上升趋势，这部分要额外加进预测里
- 如果最近 7 天没有更高，就按 0 处理，不额外加分

未来 30 天预测值：

```text
pred_30d =
0.40 * recent_30_avg * 30
+ 0.40 * same_period_avg_3y_30d
+ 0.20 * seasonality_correction
```

中文公式理解：

`未来 30 天预测值 = 最近 30 天的平均事故水平 × 40% + 过去 3 年同期 30 天平均水平 × 40% + 季节修正值 × 20%`

解释：

- 这里的意思是，未来 30 天既看最近一个月的实际情况，也看往年这个季节正常是什么水平
- 如果某个时间段本来就属于事故高发季，季节修正值就会把这个因素补进去

### 行业 / 区域维度

由于数据更稀疏，建议先按周聚合再预测：

```text
pred_30d =
0.50 * recent_4week_avg
+ 0.35 * same_period_avg_3y_4week
+ 0.15 * trend_adjustment_week
```

中文公式理解：

`行业/区域未来 30 天预测值 = 最近 4 周平均水平 × 50% + 历史同期 4 周平均水平 × 35% + 最近周度上升趋势补偿 × 15%`

解释：

- 行业和区域的数据通常比全市总量更少，所以这里不按天算，改成按周看，更稳一点
- 先看最近 4 周表现，再参考历史同期，最后补一个“最近是不是在往上走”的趋势值

### 大白话理解

这里的公式，你可以不用逐项硬看。

它真正想表达的是：

- 不能只看最近几天
- 也不能只看去年同期
- 要把“最近趋势”和“历史正常水平”一起看

如果最近 30 天已经明显比平时高，而且还在继续往上走，那预测值就会更高。

### 举个例子

比如建筑施工行业：

- 往年 3 月通常是 3 起事故
- 今年最近 30 天已经有 5 起
- 最近 4 周还在上涨

那这个模型就会判断：

`现在不是偶然高一点，而是真的在变危险。`

## 4.5 风险分值计算

先计算偏离率：

```text
deviation_ratio = (pred_value - baseline_value) / baseline_value
```

中文公式理解：

`偏离率 =（预测值 - 正常基线值）÷ 正常基线值`

解释：

- 这一步是在算：预测出来的风险，比平时高了多少
- 如果结果是 0.5，就表示比平时高了 50%

再计算标准化异常强度：

```text
z_score = (pred_value - same_period_avg_3y) / max(same_period_std_3y, min_std)
```

中文公式理解：

`异常强度 =（预测值 - 过去 3 年同期平均值）÷ max（过去 3 年同期标准差，最小波动值）`

解释：

- 这一步不是只看“高了多少”，而是看“高得是否异常”
- 如果某个行业平时波动本来就很大，那偶尔高一点不一定危险
- 如果平时很稳定，这次却突然高很多，那就更值得警惕

最终趋势模型分值建议：

```text
trend_score = min(100, 50 + 20 * deviation_ratio + 15 * z_score + major_accident_bonus)
```

中文公式理解：

`趋势模型分值 = 50 分基础分 + 偏离率得分 + 异常强度得分 + 较大事故加分，最高不超过 100 分`

更直白一点可以理解成：

`趋势模型分值 = 基础分 + “比平时高多少”的分 + “高得是否异常”的分 + “最近有没有较大事故”的分`

其中：

- `major_accident_bonus = 10`：最近 90 天若发生较大及以上事故可加分
- `min_std`：防止标准差过小导致异常放大，建议取 1 或业务最小波动值

### 大白话理解

这一段其实就是在回答一句话：

`预测出来的风险，比平时高了多少？`

它的打分逻辑可以直接理解成：

- 越比平时高，分越高
- 越是最近持续上涨，分越高
- 最近如果出过较大事故，再额外加分

也就是说，趋势模型不是“猜”，而是在看：

`现在是不是明显比正常状态更危险。`

## 4.6 分级建议

趋势模型单模型分值可映射为：

| 分值 | 等级 |
|------|------|
| < 60 | 无预警 |
| 60-69 | 蓝色 |
| 70-79 | 黄色 |
| 80-89 | 橙色 |
| >= 90 | 红色 |

说明：这是系统默认建议值，最终应开放给管理员配置。

## 4.7 原因解释生成

趋势外推法的解释模板建议固定为：

1. **历史规律**：近 3 年同期平均水平是多少，当前预测值高出多少
2. **近期变化**：最近 7 天 / 30 天是否持续上升
3. **异常事件**：近期是否发生较大事故或死亡人数异常

示例：

> 系统对建筑施工行业发出橙色预警，依据：①近 3 年同期月均事故 3.2 起，本次预测未来 30 天将达到 5.1 起，偏高 59%；②近 4 周事故数连续上升；③近 60 天该行业发生 1 起较大事故，对风险水平形成拉升。

## 4.8 适用性与限制

优点：

- 稳定、解释清楚
- 不依赖外部数据
- 易于做阈值校准

限制：

- 对突发外因不敏感
- 对低频行业和低频区域容易波动
- 主要反映“趋势升高”，不擅长识别“历史相似事件复现”

## 5. 模型二：相似度匹配法

## 5.1 模型目标

寻找“当前事故态势”在历史上最相似的时间窗口，并用这些历史相似窗口后续实际发生的事故情况，推断当前未来风险。

这是一期第二个基础模型，主要解决两个问题：

- 仅靠趋势外推法时，对“结构变化”识别不足
- 需要给出更贴近业务认知的解释，例如“当前像历史上的哪段时间”

## 5.2 核心设计思路

把每个“观测窗口”转换成一个特征向量，再去历史库里检索最相似的若干窗口。

当前窗口可以定义为：

- 最近 30 天总体态势
- 最近 4 周行业态势
- 最近 4 周区域态势

然后用这些相似窗口在其“后续 7 天 / 30 天”真实发生的事故数，推断当前未来风险。

### 大白话理解

这个模型最像人平时会说：

`现在这个情况，我以前见过，后来真的出过事。`

所以它不是只看“最近有没有上涨”，而是在看：

`当前这个状态，像不像历史上某一段危险时期。`

## 5.3 特征向量设计

### 总量维度特征

建议包含：

- 最近 7 天事故起数
- 最近 30 天事故起数
- 最近 30 天死亡人数
- 最近 4 周周度变化率
- 当前月份 / 季度
- 最近 90 天较大事故标记
- 行业集中度（前 3 行业占比）
- 区域集中度（前 3 区域占比）

### 行业维度特征

- 最近 30 天该行业事故起数
- 最近 30 天该行业死亡人数
- 该行业占全市事故比重
- 最近 4 周周度环比变化
- 历史同期偏离度
- 该行业在重点区域的集中度

### 区域维度特征

- 最近 30 天该区域事故起数
- 最近 30 天该区域死亡人数
- 该区域占全市事故比重
- 最近 4 周周度环比变化
- 历史同期偏离度
- 该区域重点行业构成占比

## 5.4 相似度计算方法

建议采用“加权标准化欧氏距离”或“加权余弦相似度”。

推荐实现：

```text
distance(X, Y) = Σ wi * |Xi - Yi| / scale_i + season_penalty
```

中文公式理解：

`相似距离 = 每个特征差异 × 对应权重，再除以这个特征平时的波动大小，最后再加上季节差异惩罚`

更直白一点可以理解成：

`相似距离 = 当前状态和历史状态“差多少”的总和`

解释：

- 这里不是算“越大越好”，而是“越小越像”
- 距离越小，表示当前状态和某段历史时期越接近
- 如果当前是 3 月，历史样本却是 8 月，就算其他数据像，也会因为季节差异被扣一点分

其中：

- `wi`：特征权重
- `scale_i`：该特征历史标准差，用于标准化
- `season_penalty`：季节惩罚项，若月份差异较大则增加惩罚

建议默认权重：

| 特征组 | 权重 |
|--------|------|
| 近期事故总量 | 0.30 |
| 死亡/重伤情况 | 0.20 |
| 上升趋势 | 0.20 |
| 行业/区域结构 | 0.20 |
| 时间季节因素 | 0.10 |

### 大白话理解

这里所谓“相似度”，你就理解成“长得像不像”。

系统会拿当前这段时间，去和历史上的很多时间段比：

- 事故数量像不像
- 死亡人数像不像
- 是不是某个行业占比变高了
- 是不是某个区域更集中了
- 当前月份和历史月份像不像

越像，说明越可能重演历史上类似的风险走势。

## 5.5 检索与预测逻辑

1. 从历史库中筛选满足最小样本条件的候选窗口
2. 计算与当前窗口的相似距离
3. 选出 Top K 相似窗口，建议 `K = 5`
4. 取这些窗口在其后续 7 天 / 30 天真实发生的事故数，做加权平均

公式示意：

```text
similar_pred = Σ (actual_future_i * similarity_weight_i) / Σ similarity_weight_i
```

中文公式理解：

`相似模型预测值 = 把“最像的几段历史时期”后续实际发生的事故数，按相似程度加权平均`

解释：

- 越像当前状态的历史时期，权重越大
- 没那么像的历史时期，权重越小
- 所以最后不是简单平均，而是“越像越有参考价值”

其中：

```text
similarity_weight_i = 1 / (distance_i + epsilon)
```

中文公式理解：

`某个历史样本的权重 = 1 ÷（它和当前状态的距离 + 很小的保护值）`

解释：

- 距离越小，权重越大
- 距离越大，权重越小
- 加这个很小的保护值，是为了防止除数为 0

### 大白话理解

这 4 步其实可以简单理解成：

1. 先找历史上最像现在的几段时间
2. 再看那几段时间后面发生了什么
3. 如果那些时期后面事故都变多了
4. 那系统就认为现在也值得警惕

### 举个例子

比如铁西区最近：

- 总量不算爆高
- 但道路运输占比在升
- 死亡人数也有点抬头

系统一查，发现这和历史上两段高风险时期很像，而且那两次后面事故都增加了。

那这个模型就会给较高分。

## 5.6 风险分值计算

```text
similar_score =
min(100, 50
+ 25 * (similar_pred / baseline_value - 1)
+ 10 * matched_case_severity
+ 10 * matched_case_consistency)
```

中文公式理解：

`相似模型分值 = 50 分基础分 + “相似历史后续风险比正常高多少”的分 + “相似案例里是否出现较大事故”的分 + “相似案例结论是否一致”的分，最高不超过 100 分`

更直白一点可以理解成：

`相似模型分值 = 基础分 + 历史相似案例风险抬升分 + 严重程度分 + 一致性分`

说明：

- `matched_case_severity`：相似历史窗口后续若曾发生较大事故，则提高分值
- `matched_case_consistency`：Top K 历史窗口若大多都指向风险升高，则提高分值

### 大白话理解

这个模型的打分逻辑可以直接理解成：

- 现在越像历史上的危险阶段，分越高
- 找到的相似历史越多、越一致，分越高
- 如果那些历史时期后面还发生过较大事故，分再往上加

## 5.7 原因解释生成

相似度匹配法的解释建议使用：

1. 与当前最相似的历史时期是什么时候
2. 当时的主要结构特征是什么
3. 该时期后续发生了什么结果

示例：

> 系统对铁西区发出黄色预警，依据：①当前态势与 2024 年 9 月上旬、2023 年 10 月中旬两个历史窗口相似度较高；②相似窗口均表现为道路运输占比上升、区域死亡人数抬升；③这两段历史窗口在后续 30 天内事故数平均增加 28%，因此判定该区域短期风险升高。

## 5.8 样本不足时的降级策略

相似度匹配法必须设置降级规则，否则行业和区域容易乱报：

- 若历史有效窗口数 `< 20`，不独立触发预警，仅作为辅助参考
- 若某行业近 3 年累计事故数 `< 10`，不输出高于黄色的预警
- 若某区域近 3 年累计事故数 `< 8`，只保留蓝/黄预警能力

## 5.9 适用性与限制

优点：

- 能识别“结构相似但趋势不明显”的风险
- 解释更贴近业务直觉
- 适合配合趋势外推法做互补

限制：

- 强依赖历史样本量
- 对新类型风险识别弱
- 数据质量差时容易匹配失真

## 6. 模型三：因素关联法

## 6.1 模型目标

结合外部环境因素，对事故风险进行多因素增强判断，重点识别“趋势尚不明显，但外部条件已显著恶化”的情形。

这是二期增强模型，主要用于：

- 节假日前专项预警
- 极端天气预警
- 开复工时段预警
- 特定行业季节性风险预警

## 6.2 设计建议

不建议二期一开始直接上复杂机器学习模型，建议分两步：

### 第一步：可解释评分卡模型

先做规则化、可解释的“因素风险评分卡”。

### 第二步：回归模型增强

当外部数据积累 1-2 年后，再升级到 Poisson 回归 / XGBoost + SHAP。

## 6.3 一阶段评分卡模型设计

### 模型结构

```text
factor_score = baseline_score + weather_score + holiday_score + load_score + hazard_score
```

中文公式理解：

`因素关联模型分值 = 基础分 + 天气因素加分 + 节假日/重大活动加分 + 生产负荷加分 + 隐患因素加分`

解释：

- 这个模型最容易理解，本质上就是“风险加分卡”
- 满足一个危险条件，就往上加一点分
- 危险条件越多，总分越高

其中：

- `baseline_score`：来自趋势外推法或历史基线
- `weather_score`：天气因素加分
- `holiday_score`：节假日、重大活动、复工复产因素加分
- `load_score`：生产负荷因素加分
- `hazard_score`：隐患/专项治理问题数量加分

### 大白话理解

这个模型最好理解。

它本质上就是一个“风险加分卡”：

- 满足一个危险条件，就加一点分
- 危险条件越多，总分越高

你可以把它理解成：

`基础分 + 天气加分 + 节假日加分 + 复工加分 + 隐患加分`

## 6.4 因子库设计

### 道路运输行业

| 因子 | 触发条件 | 建议加分 |
|------|----------|----------|
| 连续降雨 | 未来 7 天连续降雨 >= 3 天 | +8 |
| 暴雪/结冰 | 未来 7 天出现暴雪或道路结冰预警 | +15 |
| 节假日高峰 | 春节、国庆、清明等出行高峰 | +10 |
| 近 30 天较大交通事故 | 是 | +12 |

### 建筑施工行业

| 因子 | 触发条件 | 建议加分 |
|------|----------|----------|
| 大风天气 | 未来 7 天大风 >= 6 级 | +8 |
| 强降雨 | 连续降雨或暴雨 | +10 |
| 高温 / 严寒 | 极端温度持续 | +6 |
| 节后复工 | 春节后、长假后 15 天内 | +8 |
| 夜间赶工期 | 待后续数据支持 | +5 |

### 区域维度

| 因子 | 触发条件 | 建议加分 |
|------|----------|----------|
| 极端天气预警 | 区域天气红橙黄预警 | +8 至 +15 |
| 节庆活动/人流密集 | 是 | +5 至 +10 |
| 近一周隐患数量上升 | 较均值上涨 > 30% | +8 |
| 近 30 天较大事故 | 是 | +10 |

说明：

- 上述分值是建议默认值，最终需要与甲方业务专家共同校准。
- 若当前系统暂时没有隐患数据和生产负荷数据，则对应因子不启用。

### 举个例子

比如道路运输行业当前基础分已经不低了，这时又出现：

- 下周连续降雪
- 节后返程高峰
- 最近 30 天还有较大交通事故

那系统就会认为：

`虽然现在事故量还没完全爆出来，但外部条件已经很危险了。`

所以这个模型会继续把总分往上推。

## 6.5 二阶段回归增强模型

当二期积累足够数据后，建议升级为可解释回归模型：

```text
log(λ) =
β0
+ β1 * recent_accident_count
+ β2 * rain_days
+ β3 * snow_warning
+ β4 * holiday_flag
+ β5 * resume_work_flag
+ β6 * hidden_hazard_count
+ β7 * industry_exposure
```

中文公式理解：

`未来事故风险强度 = 固定基础值 + 最近事故数量影响 + 降雨影响 + 降雪影响 + 节假日影响 + 复工影响 + 隐患数量影响 + 行业暴露度影响`

解释：

- 这里虽然写成了回归公式，但你可以把它理解成“多个因素共同决定未来风险”
- 每个因素都有自己的影响大小
- 哪个因素影响大，最终就会把预测值推得更高

其中：

- `λ` 为未来 7 天 / 30 天事故期望值
- 系数 `β` 通过历史样本训练得到

优先建议：

- 总量维度：Poisson / Negative Binomial
- 行业维度：按行业分别训练，或“行业 one-hot + 因子”
- 区域维度：按区域分别训练，或“区域 one-hot + 因子”

## 6.6 因素关联法解释模板

1. 历史基线是否正常
2. 哪些外部因素推高了风险
3. 每个因素贡献了多少风险分值

示例：

> 系统对全市道路运输领域发出橙色预警，依据：①近 3 年同期未来 7 天平均事故 6.4 起，本次模型预测为 9.1 起；②未来一周连续降雪 4 天，天气因素贡献 15 分；③恰逢节后复工和返程高峰，节假日因素贡献 10 分。

## 6.7 适用性与限制

优点：

- 能提前识别外因驱动风险
- 对专项时段预警非常有效
- 解释性可以做到较强

限制：

- 强依赖外部数据质量与稳定性
- 二期建设成本较高
- 因子过多时需要持续校准

## 7. 多模型融合策略

## 7.1 融合原则

三个模型不是“谁替代谁”，而是并行计算后融合输出。

建议权重：

### 一期

```text
final_score = 0.60 * trend_score + 0.40 * similar_score
```

中文公式理解：

`一期最终分值 = 趋势模型分值 × 60% + 相似模型分值 × 40%`

解释：

- 一期更相信趋势模型，所以它占 60%
- 相似模型作为辅助判断，占 40%

### 二期

```text
final_score = 0.45 * trend_score + 0.25 * similar_score + 0.30 * factor_score
```

中文公式理解：

`二期最终分值 = 趋势模型分值 × 45% + 相似模型分值 × 25% + 因素模型分值 × 30%`

解释：

- 二期接入外部数据后，因素模型开始参与总评分
- 但它不是完全替代前两个模型，而是作为第三个判断角度加入

### 大白话理解

你可以把这一步理解成：

`3 个模型像 3 个老师，分别打分，最后系统做综合判断。`

其中：

- 趋势模型更像“看最近走势的老师”
- 相似模型更像“看历史案例的老师”
- 因素模型更像“看外部环境的老师”

一期先让前两个老师一起打分，二期再把第三个老师加进来。

## 7.2 融合修正规则

### 升级规则

- 若任一模型为红色，且另一个模型不低于黄色，则最终至少橙色
- 若趋势模型和因素模型均为橙色及以上，则直接进入高优先级预警

### 降级规则

- 若相似度模型样本不足，权重自动降为 0
- 若因素关联法外部数据缺失，权重自动降为 0
- 若多模型分歧过大，最终等级不超过黄色，并标记“低置信度”

## 7.3 置信度计算

建议按以下因子综合评估：

- 样本是否充足
- 历史基线是否稳定
- 多模型结论是否一致
- 外部数据是否完整

可映射为：

| 条件 | 置信度 |
|------|--------|
| 样本足、模型一致、数据完整 | 高 |
| 部分样本不足或模型轻度分歧 | 中 |
| 样本不足或模型明显冲突 | 低 |

## 8. 四色预警阈值建议

## 8.1 默认分值阈值

| 风险分值 | 预警等级 | 建议处置 |
|----------|----------|----------|
| < 60 | 无预警 | 仅后台记录 |
| 60-69 | 蓝色 | 关注并持续监控 |
| 70-79 | 黄色 | 加强监控，必要时提示相关处室 |
| 80-89 | 橙色 | 重点关注，建议采取预防措施 |
| >= 90 | 红色 | 立即响应，必须登记处置措施 |

### 大白话理解

颜色不是模型直接算出来的，而是系统把最后总分“翻译”成颜色：

- 蓝色：有苗头，先关注
- 黄色：风险开始明显，要加强盯
- 橙色：风险比较高，要重点处理
- 红色：风险很高，必须马上响应

## 8.2 建议防抖规则

为了防止“狼来了”效应，建议增加防抖机制：

1. 蓝色 / 黄色预警需连续 2 次计算超阈值才生成
2. 橙色 / 红色预警可单次触发
3. 同一维度同一对象在未处置前不重复生成新预警
4. 若连续 3 次计算都低于蓝色阈值，则自动转为解除状态

## 9. 模型评估与校准方案

## 9.1 上线前回测

建议使用滚动回测：

- 训练窗口：历史前 3 年
- 验证窗口：后续 6-12 个月
- 每月向前滚动一次

## 9.2 核心评估指标

| 指标 | 含义 |
|------|------|
| 预警命中率 | 发出预警后对应窗口内实际发生事故的比例 |
| 红橙预警准确率 | 高等级预警中真正高风险的比例 |
| 漏报率 | 实际高风险但未预警的比例 |
| 平均提前量 | 预警领先事故高发的天数 |
| 误报率 | 最终被标记为误报的比例 |

## 9.3 上线后校准

建议节奏：

- 第 1 个月：只观察，不大幅调阈值
- 第 2-3 个月：按误报率和漏报率微调阈值
- 第 4-6 个月：重新校准权重和因子加分

## 11. 可直接补入功能设计文档的结论

可在 [事故风险预警模块功能设计.md](c:\Users\zhaodawen\Desktop\产品工具\评估文档\事故统计分析系统-沈阳\明细文档\功能设计文档\事故风险预警模块功能设计.md) 中补充以下结论：

1. 一期预警模型采用“趋势外推法 + 相似度匹配法”双模型并行，输出统一风险分值。
2. 二期在外部数据接入后上线“因素关联法”，作为增强模型接入，不替代一期模型。
3. 预警分值统一标准化为 0-100，默认按 60/70/80/90 映射蓝黄橙红。
4. 行业和区域维度设置样本充足度门槛，不满足条件时自动降级，避免误报。
5. 所有预警必须输出结构化原因解释，至少包含“历史规律 + 近期变化 + 相似案例或外部因素”。
