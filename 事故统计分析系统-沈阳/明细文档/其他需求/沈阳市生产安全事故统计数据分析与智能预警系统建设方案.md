# 沈阳市生产安全事故统计数据分析与智能预警系统建设方案

**汇报单位**：〔乙方单位名称〕

**汇报对象**：沈阳市应急管理局法规处

**汇报日期**：2026年3月

---

## 一、背景与必要性

### 1.1 政策依据

生产安全事故统计分析与预警是各级应急管理部门的法定职责。相关政策要求如下：

| 政策文件 | 核心要求 |
|----------|----------|
| 《安全生产法》（2021年修订） | 明确应急管理部门负有对生产安全事故实施统计分析和风险预警的职责 |
| 《生产安全事故报告和调查处理条例》（国务院令第493号） | 规定事故统计上报制度，要求各级应急管理部门定期汇总、分析事故数据 |
| 应急管理部"生产安全事故统计信息直报系统"制度要求 | 全国统一推行直报制度，沈阳市已开通独立账号（sgtj2020.mem.gov.cn） |
| 等保2.0三级安全标准（公安部） | 政务数据系统安全基础要求，涵盖访问控制、操作审计、数据加密 |

### 1.2 现状与痛点

目前，沈阳市事故调查科主要依靠**人工方式**从应急管理部直报系统中提取事故数据、手工汇总、撰写统计报告，存在以下突出问题：

**（一）统计效率低，报告周期长**

月度、季度、年度统计报告均依赖人工从直报系统逐项导出数据，再手工制作图表和撰写分析文字，耗时数天，难以实现及时发布。

**（二）监管模式滞后，无预警能力**

目前只能"看到已发生的数字"，无法基于历史规律预判未来风险走势。事故数据的价值仅停留在"事后记录"层面，未能转化为"事前预警"的监管能力。

**（三）数据共享依赖人工，跨部门协同效率低**

交通局、消防救援、公安等部门在开展专项整治时，如需查阅与本部门相关的事故统计数据，须向应急管理局提出申请，再由工作人员人工提取、整理后提供，周期长、响应慢，且无标准化授权机制。

### 1.3 建设时机

当前，应急管理部直报系统已在全国推行多年，沈阳市积累了相当体量的历史事故数据，具备数据基础。与此同时，大数据分析和智能预警技术已在同类政务场景中有成熟应用，建设条件成熟。

## 二、建设目标

### 2.1 总体定位

本系统定位为应急管理部"生产安全事故统计信息直报系统"（沈阳账号）的**本地智能分析大脑**。

> 核心原则：**不改变原有直报流程**，不替代现有直报系统，仅在数据层面进行读取分析，实现两大核心转变。

### 2.2 两大核心转变

**转变一：统计报告生成从人工到自动**

系统自动从直报系统同步事故数据，一键生成月度简报、季度深度分析报告、年度白皮书三类标准格式报告，配套图表、同比/环比分析和AI文字解读，导出Word/PDF后可直接使用。

**转变二：安全监管模式从事后到事前**

基于历史事故数据规律，系统持续对全市事故态势进行分析预测，在风险升高时自动发出分级预警（红/橙/黄/蓝四色），并给出可量化的预警原因，辅助监管人员提前介入。

### 2.3 系统与直报系统的关系

```
应急管理部直报系统（国家级）
    ↓  每日自动同步（只读，不写入）
沈阳市智能分析系统（本地建设）
    ├── 自动生成统计报告（内部使用）
    ├── 智能预警推送（内部使用）
    └── 跨部门数据查询（按权限开放给交通局、消防、公安等）
```

直报系统用于事故**上报**，本系统用于数据**分析**，两者职责清晰，互不干扰。

## 三、系统功能方案

### 3.1 四大功能模块

| 模块 | 服务对象 | 核心价值 |
|------|----------|----------|
| 数据接入与治理 | 系统底层 | 每日自动同步直报数据，清洗入库，为上层功能提供数据基础 |
| 统计分析报告 | 事故调查科工作人员、局领导 | 一键生成图文并茂的统计报告，替代人工汇总 |
| 事故风险预警 | 事故调查科工作人员、局领导 | 持续监测风险态势，自动发出分级预警并附带原因说明 |
| 数据查询（外部） | 交通局、消防、公安等授权部门 | 在权限范围内自主查询关联事故数据，无需人工提取 |

系统另设**系统管理模块**，负责用户账号与权限配置、全链路操作日志留痕，满足合规要求。

### 3.2 统计分析报告模块

系统内置三种报告模板，覆盖日常汇报需要：

- **月度简报**：1页纸核心指标，适合月末快速汇总
- **季度深度分析报告**：图文并茂，含行业分布、区域分析、趋势对比
- **年度白皮书**：全维度数据汇总，适合年度总结与对外发布

**核心功能亮点：**

（1）**同比/环比自动标注**：所有统计图表自动计算与去年同期（同比）和上一统计周期（环比）的变化率，上升标红色，下降标绿色，一目了然。

（2）**AI图文智能解读**：系统为每张图表自动生成一段公文风格的文字分析，不仅展示数字，还给出背后的原因判断（如"剔除X月重大事故影响后，整体趋势保持平稳"）。

（3）**异常点自动标注**：趋势图中若某时间节点数据异常波动，系统自动标注并弹出该时间节点的事故摘要，方便溯源。

（4）**多维度下钻**：报告中点击任意行业或区域指标，即可展开该维度的事故类型、发生时段、区域分布等详细信息。

（5）**一键导出与邮件分发**：支持导出Word、PDF格式；支持预设收件人列表，一键将报告通过邮件发送给指定人员。

### 3.3 事故风险预警模块

系统提供三种预警模型，可按需配置：

| 预警模型 | 原理 | 适用场景 |
|----------|------|----------|
| 趋势外推法 | 基于历史时间序列预测未来走势 | 常规周期性风险研判 |
| 相似度匹配法 | 寻找与当前形势最相似的往年同期，参考其后续走势 | 特定季节性规律分析 |
| 因素关联法（进阶） | 结合气象、节假日等外部数据进行多因素分析 | 重大节点前的专项预警 |

预警支持三个维度，覆盖不同层级的监管需要：

- **总量预警**：预测未来一周/一月全市事故起数是否可能超过历史阈值
- **类型预警**：预测化工、建筑施工等特定行业事故风险升高
- **区域预警**：在地图上对高风险区县进行四色分级标注（红/橙/黄/蓝）

**每条预警必须附带可量化的原因说明**，例如："系统对A区发出橙色预警，依据：①A区过去三年3月份事故发生率高于全市均值32%；②近一周A区上报隐患数量增加30%；③未来一周有连续降雨天气。"系统不输出无解释的"黑盒预警"。

### 3.4 跨部门数据查询模块

为交通局、消防救援支队、公安局等有授权的外部政务部门提供自助数据查询能力，建立标准化的数据共享机制，替代当前人工提数模式。

**核心设计原则：严格权限隔离**

- 各外部部门用户只能查询与本部门监管职责相关的事故类型和区域范围（如交通局仅可查道路运输类事故）
- 查询结果仅展示聚合统计数据，不涉及事故当事人个人信息等敏感字段
- 权限配置由应急管理局系统管理员统一维护，外部部门用户不可自行修改
- 后端对每次查询请求进行权限二次校验，防止参数篡改越权

**工作流程对比：**

| 对比维度 | 当前方式 | 建设后 |
|----------|----------|--------|
| 外部部门获取数据 | 向应急管理局提出申请 → 工作人员人工提取 → 整理发送 | 外部部门用账号自主登录查询，即时获取 |
| 响应时效 | 数天 | 实时 |
| 授权机制 | 无标准化记录 | 统一权限配置，操作全程留痕 |
| 数据安全 | 依赖人工判断边界 | 系统强制权限隔离，后端双重校验 |

## 四、数据安全与合规保障

### 4.1 数据来源合规性

本系统的数据来源唯一，即应急管理部"生产安全事故统计信息直报系统"（沈阳市账号）。

- 数据访问基于沈阳市应急管理局的合法账号授权，仅读取沈阳市权限范围内的事故数据
- 系统不修改直报系统中的任何原始数据，不在原有直报流程中增加任何环节
- 本地存储的事故数据为从直报系统同步的沈阳市数据，不涉及其他省市数据

### 4.2 外部部门访问的合规机制

跨部门数据查询功能的授权机制遵循以下原则：

（1）**明确授权**：外部部门用户账号须经应急管理局管理员审批后创建，每个账号绑定具体的数据查询权限范围（事故类型白名单 + 区域范围），权限来源可追溯。

（2）**最小权限**：外部用户仅能查询与本部门职责边界匹配的数据维度，不开放超出范围的数据，不开放内部报告生成和预警功能。

（3）**数据脱敏**：查询结果仅展示聚合统计指标（起数、伤亡人数、分布占比等），不展示事故编号、当事人信息等个案敏感字段。

（4）**全程留痕**：所有外部用户的登录、查询、导出操作均记录日志，供审计查询，满足数据使用安全可追溯要求。

### 4.3 系统安全合规标准

| 合规要求 | 实现方式 |
|----------|----------|
| 操作留痕 | 所有用户（内部/外部）的登录、查询、导出、预警处置操作全部记录日志 |
| 访问控制 | 基于角色的权限体系，内部用户与外部部门用户权限严格隔离 |
| 数据加密 | 传输层HTTPS加密，存储层敏感数据加密存储 |
| 等保合规 | 系统设计满足等保2.0三级标准，可支持等保测评 |
| AI内容合规 | 报告中的AI生成文字为辅助参考，发布前须经工作人员人工审核，不自动发布 |

## 五、建设路径

### 5.1 分阶段建设建议

考虑到数据接口确认、权限清单制定等前置工作需要时间，建议分两期建设：

**一期（核心交付）：** 数据接入 + 统计分析报告 + 基础预警

- 打通与直报系统的数据同步通道
- 上线月度、季度、年度三类报告自动生成
- 上线趋势外推法和相似度匹配法两种基础预警模型
- 上线系统管理（用户权限管理、操作日志）

**二期（扩展能力）：** 跨部门查询 + 进阶预警

- 上线外部部门自助数据查询门户
- 上线因素关联法预警（结合气象、节假日等外部数据）
- 报告邮件自动分发
- 预警准确率统计与模型优化

### 5.2 启动前需确认的关键事项

以下事项涉及甲方侧的决策和配合，建议在项目启动前优先确认：

| 序号 | 确认事项 | 重要性 | 负责方 |
|------|----------|--------|--------|
| 1 | 应急管理部直报系统的数据获取方式（官方API / 账号导出 / 其他） | 🔴 最高 | 事故调查科 + 技术对接 |
| 2 | 各外部部门（交通局、消防、公安）的数据查询权限范围清单 | 🔴 最高 | 法规处牵头，协调各部门 |
| 3 | 系统部署环境（政务云 / 自有机房）及网络策略 | 🟡 高 | 信息化主管部门 |
| 4 | 报告导出格式的版式要求（是否需符合特定公文格式） | 🟡 高 | 事故调查科 |
| 5 | 邮件分发是否在政务内网中操作及邮件服务器信息 | 🟡 高 | 信息化主管部门 |
| 6 | AI文字解读的生成方式（政务内网私有模型 / 外网API） | 🟡 高 | 信息化主管部门 |

> **特别说明**：第2项（各部门权限范围清单）建议由法规处牵头，基于各部门的法定职责边界进行界定，避免后续权限争议。

## 六、预期价值

### 6.1 对应急管理局内部

- **提升效率**：月度/季度/年度统计报告从数天缩短至一键生成，工作人员从重复性数据整理工作中解放出来
- **提升质量**：报告自动生成减少人工误差，AI解读统一分析风格，同比/环比数据自动计算保证准确性
- **提升监管能力**：从"事后看数字"升级为"事前预判风险"，预警提前发出，监管介入更及时

### 6.2 对跨部门协同

- **替代人工提数**：交通局、消防、公安等部门可随时自主查询所需数据，不再占用应急管理局工作人员时间
- **建立标准化共享机制**：数据共享有明确的授权范围、访问记录和审计痕迹，规范化程度显著提升
- **支撑联合整治**：各部门掌握各自领域的事故统计数据，有助于在专项整治行动中提供数据支撑

### 6.3 对上级监管部门

- **数据可追溯**：全链路操作留痕，数据使用安全可查
- **报告质量提升**：向上级报送的统计报告格式统一、数据准确，提升专业形象
- **政策响应能力**：系统预警机制与应急管理部推动的"事前预防"监管转型方向高度契合
