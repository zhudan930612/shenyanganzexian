## 一、背景与目标

### 为什么要建知识库

当前AI课程生产质量不达标，根本原因不是模型能力不足，而是**模型没有可依赖的专业知识体系**。模型在缺少知识支撑的情况下只能依赖训练数据生成内容，导致法规引用错误、合规表述违规、案例造假等核心问题。

知识库建设是解决内容质量问题的前提条件，其他优化（工作流、提示词）建立在知识库之上。

### 建设目标

| 时间范围 | 目标 |
|----------|------|
| 短期（3个月内） | 合规表述违规问题减少80%；法规引用准确率提升至90%以上；有效素材占比提升50%以上 |
| 长期 | 建立可持续维护的保险专业知识体系，支撑AI稳定输出达到人工终审可用的内容 |

### 边界说明

**知识库可以解决**：法规条文引用错误、合规表述违规、专业术语混用、案例内容造假

**知识库不能完全解决**（需配合工作流优化）：内容结构松散、教学设计针对性差、分析深度不足

> 保险内容涉及合规，知识库可大幅降低错误率，但人工终审仍为必要环节。目标是降低人工介入的成本和深度，不是完全消除。

## 三、填写通用规范

以下规范适用于所有知识库，内容团队必须严格遵守。

### 格式要求

- 所有文件使用 **Excel 格式**，不接受 Word 或 PDF
- 每行只记录一条信息，**不允许合并单元格**
- 字段名称不得修改，严格按照模板填写
- 标签类字段只能从规定选项中选择，**不得自创**

### 质量红线

- **法规条文库**：原文不得修改，必须与官方文件一字不差
- **司法案例库**：案件编号必须真实，不得填写虚构案例
- **合规红线库**：合规表述必须经合规部门审核确认后录入

### 不确定内容的处理

- 定义不确定 → 标注"待确认"，不得自行填写
- 找不到依据 → 依据字段填"暂无"，不得编造
- 案例来源不明 → 不录入，**宁缺勿滥**

## 七、KB4：产品规则库

### 定位

### 建设原则

不上传产品说明书原文，只录入从条款中提炼的结构化规则。原文太长，检索时会召回大量无关内容。

### 每个险种需覆盖的规则维度

```

## 八、KB5：司法案例库

### 定位

提供真实的保险司法案例，彻底解决案例造假问题。

> **核心约束**：检索不到真实案例时，AI必须输出"暂无匹配案例"，绝不允许生成案例。此约束已内置于工作流节点，不可删除。

### 建设节奏

```
第13-14周：内部案例入库
  公司历史理赔纠纷案例（脱敏处理后入库）
  目标：50-100个，覆盖主要险种

第15-20周：外部典型案例
  北大法宝或威科先行按保险案由筛选导出
  目标：每月新增30-50个

持续补充：每月例行新增10-20个
```

### 筛选标准

**优先收录**：中级法院及以上判决 / 近5年判决 / 有明确争议焦点和清晰法院意见的案例

**不收录**：基层法院调解结案 / 10年以上旧案 / 案情模糊、判决理由不清晰的案例

### 案例分类体系

```
人身险
├── 理赔拒付纠纷
├── 如实告知义务

健康险

财产险

| 最终召回数量 | 5 |

### 验收标准

用10个案例场景测试，能召回真实案例且无编造内容即为合格。

## 十、验证机制

每个知识库建完后，必须完成验证再接入工作流，**验证不合格不得接入**。

| 知识库 | 验证方法 | 合格标准 |
|--------|----------|----------|
| 合规红线库 | 10个典型违规场景测试 | 每个场景召回3条以上相关红线 |
| 核心概念库 | 随机抽取20个概念测试 | 定义准确率≥90% |
| 法规条文库 | 20个法规引用场景测试 | 条文准确率≥95% |
| 产品规则库 | 20个规则场景测试 | 描述准确率≥90% |
| 司法案例库 | 10个案例场景测试 | 召回真实案例，无编造内容 |
